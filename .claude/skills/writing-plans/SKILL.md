# Написание планов (telegram-sub-bot)

## Обзор

Пиши подробный план внедрения так, будто его будет выполнять разработчик, который **не знает наш бот**: какой файл трогать, какой код писать, как протестировать. Разбивай работу на мелкие шаги (2–5 минут каждый). Принципы: **DRY, YAGNI, TDD, частые коммиты**.

**Анонс в начале:** «Использую скилл writing-plans — составляю план под задачу бота подписок.»

**Куда сохранять план:** `docs/plans/YYYY-MM-DD-<короткое-имя>.md`.

---

## Контекст проекта (не повторять в каждом плане, но иметь в виду)

- Стек: Python 3.11+, aiogram 3.x, asyncpg, APScheduler, aiohttp.
- БД: PostgreSQL, таблицы `users` и `payments` (см. `app/db/migrations.sql`).
- Оплата: CryptoPay (@CryptoBot) через `aiocryptopay`, только USDT. Подпись HMAC-SHA256 = `HMAC(SHA256(app_token), body_raw)`, проверяется встроенно (`crypto.check_signature` / `crypto.get_updates`).
- Тарифы: `tariff_3d` = 11$/3д, `tariff_7d` = 21$/7д, `tariff_30d` = 60$/30д.
- 4 приватных чата, бот — админ с правами invite/restrict.
- Деплой — webhook (polling не используется).

Опираясь на это, при планировании задачи **выбирай правильный слой** из структуры (`app/bot/`, `app/payments/`, `app/db/`, `app/chats/`, `app/scheduler/`) и не создавай новых, если можно расширить существующий.

## Scope-проверка

Перед началом проверь, не является ли задача «пакетом»: например, «добавить реферальную программу» = изменения в БД + хендлеры + крон + тексты. Если задач реально несколько независимых — предложи разделить на отдельные планы. Каждый план должен давать рабочий, тестируемый результат сам по себе.

Типичные самостоятельные единицы в этом проекте:
- новая команда/callback в боте,
- изменение в интеграции CryptoPay (новое поле инвойса, дополнительный `update_type`),
- новая крон-задача,
- миграция БД + изменения на её основе,
- работа с чатами (добавление 5-го чата, смена прав, массовый kick).

Перед задачами опиши **какие файлы** будут затронуты и **за что каждый отвечает**. Типовые зоны:

## Гранулярность шагов

Каждый шаг — одно действие (2–5 минут):

- «Написать падающий тест» — шаг
- «Убедиться, что тест падает» — шаг
- «Реализовать минимум, чтобы тест прошёл» — шаг
- «Запустить тесты» — шаг
- «Коммит» — шаг

## Структура задачи

````markdown
### Задача N: [название]

**Файлы:**
- Создать: `app/path/to/file.py`
- Изменить: `app/path/to/existing.py:строки 40-65`
- Тест: `tests/path/to/test_something.py`

- [ ] **Шаг 1. Написать падающий тест**

```python
# tests/payments/test_payload_encoding.py
from app.payments.cryptopay import encode_payload, decode_payload

def test_payload_roundtrip():
    assert decode_payload(encode_payload(123, "tariff_7d")) == (123, "tariff_7d")
```

- [ ] **Шаг 2. Запустить тест — убедиться, что падает**

Команда: `pytest tests/payments/test_payload_encoding.py -v`
Ожидаемо: FAIL (функции ещё не реализованы / ImportError).

- [ ] **Шаг 3. Реализовать минимум**

```python
# app/payments/cryptopay.py (фрагмент)
def encode_payload(telegram_id: int, plan_key: str) -> str:
    return f"{telegram_id}:{plan_key}"

def decode_payload(payload: str) -> tuple[int, str]:
    tg_id, plan_key = payload.split(":", 1)
    return int(tg_id), plan_key
```

- [ ] **Шаг 4. Запустить тест — убедиться, что прошёл**

Команда: `pytest tests/payments/test_payload_encoding.py -v`
Ожидаемо: PASS.

- [ ] **Шаг 5. Коммит**

```bash
git add app/payments/cryptopay.py tests/payments/test_payload_encoding.py
git commit -m "feat(payments): encode/decode payload для CryptoPay"
```
````

## Никаких заглушек

Эти формулировки в нашем плане — **провал плана**, не пиши их:

- «TBD», «доработать позже», «заполнить детали»
- «Добавить обработку ошибок» / «валидацию» / «edge case» (без конкретики — какой Exception, какой ответ)
- «Написать тесты на вышесказанное» без самого кода теста
- «Аналогично Задаче N» (повторяй код — исполнитель может читать задачи не подряд)
- Шаги, описывающие «что сделать», но не показывающие как (для кода всегда нужен код-блок)
- Упоминание функций/методов/полей БД, не определённых ни в одной задаче плана

## Чеклисты, специфичные для этого проекта

Встраивай в задачи там, где применимо:

### Если задача трогает оплату
- [ ] Webhook CryptoPay зарегистрирован через `crypto.get_updates` (встроенная HMAC-SHA256 верификация); ручной проверки подписи писать не надо.
- [ ] Маршрут `/cryptopay/webhook` регистрируется напрямую, **без middleware**, которое парсит/пересериализует body — иначе сигнатура сломается.
- [ ] Идемпотентность: `SELECT … FOR UPDATE` на `payments` + ранний return, если `status == 'paid'`.
- [ ] `payment_id` (=`str(invoice_id)`) хранится с `UNIQUE`.
- [ ] `invoice.amount` из webhook сверяется с `payments.amount`; при несовпадении — `log.error` и не зачислять.
- [ ] Webhook отвечает `200` даже на повторную доставку уже обработанного инвойса.

### Если задача трогает подписку
- [ ] Продление суммируется: `max(now, paid_until) + days`, не обнуляет.
- [ ] `status` и `paid_until` обновляются в одном UPDATE.

### Если задача трогает инвайты/чаты
- [ ] `create_chat_invite_link` всегда c `member_limit=1` и `expire_date = now + 1ч`.
- [ ] При kick используется пара `ban_chat_member` + `unban_chat_member` (чтобы пользователь мог вернуться после новой оплаты).
- [ ] Бот имеет нужные права во всех 4 чатах из `settings.chat_ids`.

### Если задача трогает крон
- [ ] Задание идемпотентно на случай двойного тика или двух инстансов.
- [ ] Окна по времени не приводят к двойной отправке напоминаний.
- [ ] Исключения в одной итерации не ломают цикл по пользователям.

### Если задача трогает БД
- [ ] SQL — в `app/db/migrations.sql` и/или `queries.py`; не инлайнится в хендлерах.
- [ ] Миграция идемпотентна (`IF NOT EXISTS`).
- [ ] Все `datetime` — `TIMESTAMPTZ` в UTC.
